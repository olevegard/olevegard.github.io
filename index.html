    An attempt at a DX Ball clone

    Project maintained by olevegard Hosted on GitHub Pages — Theme by mattgraham

    Hello and welcome to this weird place where I am writing to none other than myself!

    First of all, I'd like to say that the name is not final, I just haven't figured out a clever name for it yet. Maybe I never will.

    As you probably have guessed by now, this game draws a lot of inspiration from DX Ball. The reason I choose to do a DX Ball remake is that it's quite easy to make. I have tried making more complex games earlier, and failed. So I think DX Ball is a fitting one-person challenge.

    Technology

    As for technology, this game will be written almost entirely in C++11¹. The code is to remain platform-independent².

    When it comes to third party libraries, only SDL2.0 will be used. With the possible addition of OpenGL later on. But since SDL2.0 already uses OpenGL, it might not be necessary.

    Coding standards

    The coding standards at this project is yet to be properly defined. However, there are a few rules.

    Stylistic

    • Always put angle brackets ( { and } )
    • Function names starts with uppercase letters and uses camel case. Underscore is permitted in some cases.
    • Variable names start with a lowercase letter, and uses camel case. Underscore is permitted in some cases.
    • (System³) Hungarian notation is not permitted.

    General

    • The code is to be compiled with ./buildDXB.h. ANY warnings that occurs are to be fixed properly.
    • As far as possible, prevent the use of this to assign variables etc.
    • People are permitted to kick anyone who uses a macro
    • Cost are to be applied to variables passed by reference and functions whenever possible.
    • Functions are not to be longer than one screen length unless you have a really good reason.
    • Use proper names for variables and functions. Long names is not an issue, unless it's approaching 80 chars
    • DON'T REPEAT YOURSELF.
    • Keep things simple, don't rely on clever/ heavily optimized functions.
    • Rather than writing own code, functionality in the standard C++ library should be used.

    Following the above rules, there will be more code to write. But your code will require less comments and will be easier to understand and debug. ( Anyone who claims they don't produce bugs are liars. )

    ¹ If the game is ever ported to work on iPhone/iPad, the code might have to be ported to Objective C++(11)
    ² As of now, the code only works on Linux, this is due to lack of timing functions. But it'll be easy to port the Timer class to Windows
    ³ http://en.wikipedia.org/wiki/Hungarian_notation#Systems_vs._Apps_Hungarian.
